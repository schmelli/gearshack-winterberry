# Deployment Guide

Complete guide to deploying Gearshack Winterberry to production on Vercel.

## Overview

**Hosting**: Vercel (Edge Network)
**Database**: Supabase (PostgreSQL)
**CDN**: Cloudinary (images)
**Monitoring**: Sentry + Vercel Analytics

**Deployment Flow**:
```
git push → development
  ↓
Vercel Preview Deploy (automatic)
  ↓
Manual Review & Testing
  ↓
Merge to main → Production Deploy
```

---

## Prerequisites

### Accounts Required

1. **Vercel Account** (https://vercel.com)
   - Connected to GitHub
   - Hobby or Pro plan

2. **Supabase Project** (https://supabase.com)
   - PostgreSQL database
   - Auth configured
   - Storage enabled

3. **Cloudinary Account** (https://cloudinary.com)
   - Cloud name
   - API credentials

4. **Sentry Project** (https://sentry.io)
   - DSN for error tracking
   - Auth token for source maps

## Database Migrations

### Apply Migrations to Production

**Option A: Supabase Dashboard** (Recommended)

1. Go to [Supabase Dashboard](https://supabase.com/dashboard)
2. Select your project
3. Database → Migrations
4. Copy migration SQL
5. Run in SQL Editor
6. Verify tables created

**Option B: Supabase CLI**

```bash
# Set project
supabase link --project-ref your-project-ref

# Push migrations
supabase db push

# Verify
supabase db diff
```

### Migration Workflow

1. **Develop locally** with local Supabase
2. **Test migration** on local DB
3. **Commit migration file** to git
4. **Deploy code** to Vercel
5. **Apply migration** to production DB (manual step)

**Important**: Always apply migrations BEFORE deploying code that depends on them.

## Rollback Strategy

### Instant Rollback (Vercel)

**If production breaks**:

1. Go to Vercel Dashboard
2. Deployments → Select previous working deployment
3. Click "Promote to Production"
4. **Duration**: 30 seconds

**This rolls back**:
- Application code
- Static assets
- Edge functions

**This does NOT roll back**:
- Database changes (requires manual rollback)
- Environment variables

### Database Rollback

**Manually revert migrations**:

```bash
# Local: Test rollback SQL
supabase db reset

# Production: Run rollback in SQL Editor
DROP TABLE IF EXISTS new_table;
ALTER TABLE old_table DROP COLUMN IF EXISTS new_column;
```

**Best Practice**: Always have rollback SQL ready before applying migration.

## Performance Optimization

### Edge Caching

**Vercel automatically caches**:
- Static assets (images, fonts, CSS, JS)
- Static pages (if no dynamic data)
- API responses (with `Cache-Control` headers)

**Configure caching**:
```typescript
// API Route
export const revalidate = 3600; // Cache for 1 hour

// Or manually:
return new Response(data, {
  headers: {
    'Cache-Control': 'public, max-age=3600, s-maxage=3600',
  },
});
```

### Image Optimization

**Next.js Image Optimization**:
- Automatic WebP conversion
- Lazy loading
- Responsive sizes
- CDN delivery

**Already configured** via `next/image` component.

### Database Optimization

**Connection Pooling**: Use `DATABASE_URL` with pooler (`:6543`)
**Indexes**: Ensure all foreign keys indexed
**Query Optimization**: Use `EXPLAIN ANALYZE` for slow queries

### Bundle Size

**Check bundle**:
```bash
ANALYZE=true npm run build
```

**Optimize**:
- Lazy load heavy components
- Tree-shake unused exports
- Use dynamic imports
- Minimize dependencies

## Scaling Considerations

### Current Limits

- **Vercel**: 100GB bandwidth/month (Hobby), 1TB (Pro)
- **Supabase**: 500MB database (Free), 8GB (Pro)
- **AI Gateway**: Pay-per-use (no hard limit)
- **Cloudinary**: 25GB storage/bandwidth (Free), 300GB (Plus)

### Scaling Database

**When to upgrade**:
- Database > 80% of plan limit
- Connection pool exhausted
- Slow queries (> 1s)

**How to scale**:
1. Upgrade Supabase plan
2. Optimize queries (add indexes)
3. Implement caching (Redis)

### Scaling Serverless

**Vercel auto-scales** serverless functions:
- Cold starts: ~500ms (first request)
- Warm starts: ~50ms
- Concurrent executions: Unlimited (Hobby: 10, Pro: 100)

**If hitting limits**: Upgrade to Pro plan.

### Scaling AI Costs

**Monitor token usage**:
```bash
# Check Mastra metrics
curl https://gearshack.app/api/mastra/metrics
```

**Optimization**:
- Observational Memory (30-50% reduction)
- Use cheaper models (Gemini Flash for Observer)
- Implement per-user rate limiting

## Troubleshooting Deployments

### Build Fails

**Error**: `npm ERR! code ELIFECYCLE`

**Fix**:
1. Check build logs in Vercel
2. Run `npm run build` locally
3. Fix TypeScript errors
4. Push fix

### Environment Variable Missing

**Error**: `AI_GATEWAY_KEY is required`

**Fix**:
1. Go to Vercel → Settings → Environment Variables
2. Add missing variable
3. Redeploy (Deployments → click "Redeploy")

### Database Connection Error

**Error**: `Failed to connect to database`

**Fix**:
1. Check DATABASE_URL format (must be pooler URL with `:6543`)
2. Verify Supabase project not paused
3. Check IP allowlist (add `0.0.0.0/0` or Vercel IPs)

### Slow Response Times

**Issue**: Pages load slowly

**Fix**:
1. Check Vercel Analytics → identify slow routes
2. Add caching headers
3. Optimize database queries
4. Use Edge functions for latency-sensitive ops

### Out of Memory

**Error**: `JavaScript heap out of memory`

**Fix**:
1. Increase Node.js memory in build settings
2. Optimize dependencies (remove unused)
3. Use dynamic imports

## Disaster Recovery

### Backup Strategy

**Database**:
- Supabase: Daily automatic backups (retained 7 days on Pro)
- Manual: Export via `pg_dump` weekly

**Code**:
- Git history (permanent)
- Vercel deployment history (permanent)

**Media**:
- Cloudinary: Automatic backups (contact support)

### Recovery Scenarios

**Scenario 1: Bad Deployment**
→ Rollback via Vercel (30 seconds)

**Scenario 2: Database Corruption**
→ Restore from Supabase backup (1 hour)

**Scenario 3: Complete Failure**
→ Redeploy from Git + restore DB (2 hours)

**Scenario 4: Data Loss**
→ Restore from weekly backup (RTO: 4 hours, RPO: 1 week)
